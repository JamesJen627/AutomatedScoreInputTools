# 第十章 Cursor 开发规范（Cursor Development Specification）

Version：V1.0

---

# 10.1 目标

本章用于规范 Cursor 在整个项目开发过程中的行为。

所有 AI 生成代码必须遵循本章要求。

目的：

- 保证代码一致性
- 避免重复开发
- 保持架构稳定
- 保证后期易维护

---

# 10.2 开发原则

Cursor 开发必须遵循：

1. 先分析，再开发。
2. 优先复用已有代码。
3. 不允许重复实现相同功能。
4. 每个模块职责单一。
5. 业务逻辑不得写入 UI。
6. 所有公共逻辑统一封装。
7. 所有修改不得破坏已有功能。

---

# 10.3 文件创建规范

新增文件前必须：

① 检查是否已有相同功能。

② 检查是否已有类似模块。

③ 判断是否能够复用。

只有确认不存在时：

允许创建新文件。

禁止：

```text
utils1.ts

utils2.ts

utils_new.ts

helper.ts

helper2.ts

```

类似重复文件。

---

# 10.4 修改原则

优先：

修改已有模块。

禁止：

复制整个模块重新开发。

例如：

已有：

ScoringEngine

不得新增：

```text
ScoringEngineNew

ScoringEngineV2

ScoreCalculator

```

除非架构设计明确要求。

---

# 10.5 命名规范

所有名称采用统一英文。

类：

PascalCase

例如：

```text
ScoreRuleManager

```

函数：

camelCase

例如：

```text
calculateTotalScore()

```

常量：

UPPER_SNAKE_CASE

例如：

```text
DEFAULT_EXPORT_PATH

```

文件：

kebab-case

例如：

```text
score-rule-manager.ts

```

---

# 10.6 注释规范

所有公共类：

必须包含说明。

所有公共方法：

必须说明：

- 功能
- 参数
- 返回值

复杂算法：

必须说明计算思路。

禁止：

无意义注释。

例如：

```text
// add

```

---

# 10.7 类型规范

禁止使用：

```typescript
any

```

所有对象：

必须定义 Interface 或 Type。

禁止隐式类型。

---

# 10.8 业务逻辑规范

业务逻辑必须放入：

Domain Layer。

禁止：

React 页面

Electron 页面

Vue 页面

直接计算：

总成绩。

所有计算：

统一调用：

Scoring Engine。

---

# 10.9 Excel 操作规范

所有 Excel 操作统一由：

Excel Parser

Export Engine

负责。

禁止：

其它模块直接读取 Excel。

---

# 10.10 配置规范

所有配置：

统一读取：

Configuration Manager。

禁止：

直接读取：

config.json。

---

# 10.11 日志规范

所有日志：

统一调用：

Log Manager。

禁止：

```typescript
console.log()

```

作为正式日志。

调试代码上线前必须移除。

---

# 10.12 错误处理规范

所有异常：

必须：

- 捕获
- 分类
- 记录日志
- 返回错误对象

禁止：

空的：

```typescript
catch {}

```

禁止：

吞掉异常。

---

# 10.13 重构规范

Cursor 重构代码时：

不得修改：

业务行为。

必须保证：

输出结果一致。

所有重构：

必须通过已有测试。

---

# 10.14 单元测试规范

新增业务逻辑：

必须同时新增测试。

测试覆盖：

- 正常流程
- 边界值
- 异常值

评分引擎覆盖率：

100%。

---

# 10.15 性能规范

不得：

重复读取 Excel。

不得：

重复解析评分标准。

不得：

重复创建对象。

优先：

缓存。

---

# 10.16 AI 行为约束

Cursor 在开发过程中必须遵循以下规则：

1. 修改代码前先阅读相关模块。
2. 不得猜测业务规则。
3. 不得自行修改评分算法。
4. 不得新增未定义功能。
5. 不得删除已有接口。
6. 不得修改数据模型。
7. 不得修改 PRD 定义。

若发现需求冲突：

必须停止开发。

等待人工确认。

---

# 10.17 代码质量要求

所有代码必须满足：

- 无编译错误
- 无 ESLint 错误
- 无 TypeScript 类型错误
- 无未使用变量
- 无死代码
- 无重复代码

---

# 10.18 Git 提交规范

建议提交格式：

```text
feat: 新增评分标准管理

fix: 修复800米评分边界问题

refactor: 重构Excel解析模块

docs: 更新PRD

test: 增加评分引擎测试

style: 调整UI布局

```

---

# 10.19 AI 自检清单

Cursor 在完成每个任务后，应自动检查：

□ 是否符合 PRD

□ 是否复用已有代码

□ 是否新增重复逻辑

□ 是否影响评分引擎

□ 是否更新测试

□ 是否更新文档

□ 是否通过编译

全部通过后方可结束任务。

---

# 10.20 开发红线

以下行为严禁出现：

- 硬编码评分数据。
- 硬编码 Excel 表头。
- 修改评分算法而不更新文档。
- 绕过数据校验直接计算。
- 绕过审核直接导出。
- 直接修改原始 Excel 文件。
- 使用未经定义的数据结构。
- 为解决局部问题破坏整体架构。
